# DevOps Planning Notes

This repository uses the `DevOps` directory as the local planning area for Azure-DevOps-style work preparation.

## Working Model

- Hierarchy: `Epic -> Feature -> Backlog Item -> Task`
- Team terminology prefers `Backlog Item` over `Story`
- Planning files are drafts until they are copied into Azure DevOps
- Each planning file should include scope, assumptions, open questions, and next refinement steps

## File and Folder Conventions

- Root planning directory: `DevOps`
- Main folders:
  - `DevOps/Epics`
  - `DevOps/Features`
  - `DevOps/BacklogItems`
  - `DevOps/Tasks`
- File naming pattern: `<Id>-<Short-Title>.md`
- Use stable IDs where available, for example Azure DevOps work item numbers

## Planning Rules

- Keep one primary planning item per file
- Record assumptions explicitly instead of hiding uncertainty
- Prefer thin backlog items that can later be refined into implementation tasks
- Link child items to their parent item by ID and title
- Mark unknowns as open questions instead of inventing detail
- For each major coding task, create a dedicated test-task item

## Refinement Flow

1. Define the epic goal and boundaries
2. Split the epic into feature candidates
3. Derive backlog items with visible user or process value
4. Add tasks only when implementation planning starts
5. Revisit assumptions and open questions after each refinement pass

## Confirmed Project Decisions

- The first UI implementation targets `WPF`.
- A second implementation step should target `Avalonia UI`.
- The preferred architecture is a shared project for models, services, and view-models so both UI stacks can reuse the same application-facing state and logic where practical.
- The first delivery increment uses local unauthenticated `HTTP` access to the inverter.
- Remote access is expected to happen through a separately established `VPN` tunnel so the inverter remains logically local to the application.
- The first delivery increment uses a fixed refresh cadence of `5` seconds.
- Configurable polling is deferred to a later increment.

## Current Implementation Snapshot

- Shared application layer: `FroniusMonitor.Core`
  - Normalized Fronius models for inverter, battery, and site power flow
  - HTTP client prefers `solar_api/v1/GetPowerFlowRealtimeData.fcgi?Scope=System` and falls back to `/status/powerflow` for compatibility
  - Tolerant payload parser for both legacy and nested Fronius payload structures
  - The live snapshot now also carries `Head.Status`, inverter `Battery_Mode`, site `Mode`, `BackupMode`, and `Meter_Location`
  - Shared `FroniusDashboardViewModel` with a fixed `5` second refresh cadence and editable host binding
- Console host: `FroniusJson`
  - Refactored to consume the shared snapshot service instead of direct device parsing
  - Supports overriding the inverter host through the first command-line argument
- First desktop UI host: `FroniusMonitor.Wpf`
  - Uses dependency injection plus a WPF dispatcher adapter
  - Presents a modern dashboard for PV, load, inverter, battery, and grid exchange
- Automated validation: `FroniusMonitor.Core.Tests`
  - Covers payload parsing, dashboard refresh behavior, endpoint-binding behavior, and API fallback behavior

## Latest WPF Dashboard Notes

- The dark theme now uses explicit primary and secondary text styles to avoid unreadable dark-on-dark text.
- The main dashboard content now scrolls so card content is no longer clipped by the window edges on smaller visible areas.
- A graphical power-distribution section has been added to the `WPF` dashboard and uses the current live bindings as the first visual flow overview.
- The power-distribution view now uses component-specific visuals: `PV` yellow with a sun symbol, `Battery` green with an SOC indicator, `Home` blue, and `Grid` white.
- The central transfer point remains implicit; moving power packets communicate flow instead of separate arrow shapes.
- Packet sizing now follows a logarithmic scale so lower power levels remain visible in the diagram.
- The latest diagram refinement places the component nodes in a single top row and draws the active energy exchange underneath as arcs between source and target.
- Active arcs now emit multiple equally spaced packets that fade at the destination rather than scaling packet count by power.
- The power-distribution diagram is now hosted in its own `WPF` widget to keep `MainWindow` cleaner and make future visual refinement easier.
- Packet motion now follows the visible arc geometry directly instead of approximating the path with separate X/Y keyframes.
