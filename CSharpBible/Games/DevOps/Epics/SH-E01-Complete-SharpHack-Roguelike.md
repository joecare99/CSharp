# SH-E01 - Complete SharpHack as a Playable Roguelike

## Work Item Type
Epic

## Goal
Deliver a complete end-to-end SharpHack game experience with a clear objective, reliable progression, meaningful player choices, durable persistence, polished play surfaces, automated validation, and release readiness.

## Scope
- Complete the core run lifecycle from new game to death, victory, and restart.
- Add durable save/load support.
- Expand gameplay content and progression.
- Improve combat, enemy AI, and magic systems.
- Stabilize generated dungeons and world structure.
- Polish terminal, console, WPF, and future 3D user experiences according to staged UI priorities.
- Evolve server support toward a game server for terminal-based single-player and multiplayer sessions.
- Add release-quality testing, diagnostics, packaging, and documentation.

## Assumptions
- SharpHack remains a single-player roguelike/dungeon crawler unless later product decisions change the direction.
- UI targets can include terminal, console, WPF top view, first-person 3D, and fully immersive 3D.
- Shared library multi-targeting is preserved unless a dedicated technical decision narrows it.
- MSTest, NSubstitute, CommunityToolkit.Mvvm, and dependency injection remain preferred defaults.
- UI-facing strings and summary formatting stay in ViewModel/UI layers where practical.

## Product Decisions
- The final victory condition is obtaining the Amulet of JoCarneer.
- Both non-permadeath/reloadable and permadeath run variants should be supported.
- Permadeath replay can reuse the same dungeon with a new character.
- Permadeath replay should place a gravestone at the previous death location and may preserve part of the previous inventory if items have not rotted or been stolen.
- The save format should be compressed JSON with recovery support.
- Server scope includes terminal-based players and should support single-player and multiplayer modes over time.
- UI polish should prefer 3D first if practical, while keeping gameplay engine and ViewModels UI-neutral.
- Future 3D experiences should start with a hybrid grid-gameplay plus tile-to-3D-extrusion renderer and evolve toward prefab/mesh mapping.
- OpenGL is the initial 3D rendering preference, with a final first-stack decision to be made after checking practical .NET/WPF integration options.
- Decoration should be generated by the engine as world/gameplay data and remain available to non-3D variants.
- Decoration should start as visually decorative, but the model should allow later NetHack-like interaction with many or most objects.
- Level generation should assign room functions that influence decoration and gameplay, including armory, dining hall, pantry, serving room, kitchen, reception, battlement/wall-walk, shops, and blacksmiths.
- Initial room-function selection can be creative and code-first.

## Definition of Done
- A player can start a new run, learn controls, explore multiple levels, fight enemies, collect and manage items, save/load progress, die, win, and restart.
- Core gameplay rules are covered by deterministic MSTest tests.
- Generated maps satisfy documented invariants across representative seeds.
- Console and the selected WPF UI share core setup and gameplay services.
- Persistence is durable and versioned.
- Release documentation explains installation, controls, objective, save/load, and troubleshooting.
- Relevant builds and tests pass, ignoring only repository-approved unsupported older-framework warnings.
