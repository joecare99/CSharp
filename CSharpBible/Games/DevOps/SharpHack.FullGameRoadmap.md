# SharpHack Full Game Roadmap

## Goal
Plan the next work needed to evolve `SharpHack` from the current playable prototype into a complete roguelike game with a stable game loop, durable progression, polished UI surfaces, automated tests, and release readiness.

## Current State Summary
The current SharpHack codebase already contains a solid prototype foundation:

- `SharpHack.Base` defines core map, tile, creature, item, and contract abstractions.
- `SharpHack.Engine` coordinates sessions, field of view, stairs, level transitions, doors, pickup, equipment, combat delegation, enemy turns, and pathfinding helpers.
- `SharpHack.LevelGen` provides simple and BSP-based map generation, including room/maze/door experiments.
- `SharpHack.Combat` implements a minimal attack/defense damage model.
- `SharpHack.AI` implements direct step-toward-player enemy movement.
- `SharpHack.Persist` has in-memory level persistence and a not-yet-implemented durable `GamePersist`.
- `SharpHack.ViewModel` provides CommunityToolkit.Mvvm-based view models for tile and layered rendering.
- `SharpHack.Console`, `SharpHack.WPF2D`, `SharpHack.View.FirstPerson`, `SharpHack.Terminal`, and `SharpHack.Server` provide multiple presentation and remote-play entry points.
- MSTest projects already cover core map/creature/item behavior, combat, AI, field of view, level generation, level transitions, and first-person view-model behavior.

## Observed Gaps
The codebase is playable, but a complete game still needs focused productization work:

1. The game loop has no explicit win, lose, restart, or meta-progression rules.
2. Durable save/load is missing because `SharpHack.Persist.GamePersist` throws `NotImplementedException`.
3. Player death handling is incomplete; enemies can reduce player HP but the session does not end cleanly.
4. Level progression has no bounded depth, boss/finale, quest goal, scoring, or victory condition.
5. Enemy and item content is hard-coded to goblins, swords, and leather armor.
6. Creature, item, and encounter scaling are not yet data-driven.
7. Inventory supports pickup and auto-equip, but lacks explicit equip, drop, inspect, consumables, stack handling, and container interaction.
8. Combat is deterministic and shallow; no miss/critical effects, damage ranges, status effects, resistances, or clear combat result model exists.
9. AI does not use full pathfinding, sight, hearing, alert states, fleeing, ranged behavior, or door interaction.
10. Magic has a project shell but no implemented spells or gameplay integration.
11. WPF and console UI expose basic gameplay, but menus, save/load UI, key binding help, death/victory screens, settings, and accessibility polish are missing.
12. Multiple UI entry points need a shared application-service layer to avoid divergent setup and gameplay wiring.
13. Tests cover prototype behavior but not complete game rules, save compatibility, UI-facing workflows, or long-run generated-map invariants.
14. Release readiness is missing: deterministic smoke tests, packaging, versioning, user documentation, and crash diagnostics are not yet planned.

## Assumptions

- The intended primary genre is a single-player roguelike/dungeon crawler.
- UI targets remain intentionally broad: terminal, console, WPF top view, WPF/3D first-person, and potentially fully immersive 3D views can all be considered after engine stability.
- The current .NET multi-targeting strategy should be preserved for shared libraries unless a later technical decision narrows the target set.
- `MSTest`, `NSubstitute`, `CommunityToolkit.Mvvm`, and dependency injection remain the preferred implementation stack.
- UI-facing strings and summary formatting should stay in the ViewModel/UI layer where practical.

## Product Decisions

- The final victory condition is obtaining the Amulet of JoCarneer.
- Both death/run variants should be supported: non-permadeath/reloadable play and permadeath.
- In permadeath mode, a new character can replay the same dungeon layout after death.
- Permadeath replay should place a gravestone at the previous death location.
- Some previous-character items can remain at the death location if they have not rotted or been stolen.
- The durable save format should be compressed JSON with recovery support.
- `SharpHack.Server` should evolve toward a game server for terminal-based players, supporting single-player and multiplayer modes.
- UI polish should prefer 3D first if practical, while keeping the engine UI-neutral.
- Future 3D views should use a hybrid approach first: existing grid-based gameplay remains authoritative, with tile-to-3D extrusion as the first renderer and later prefab/mesh mapping.
- OpenGL is the initial rendering-technology preference, but the final first 3D stack should be selected after comparing practical .NET/WPF integration options.
- Decoration is gameplay/world data generated by the engine, not just view-local 3D dressing, and must remain available to other game variants and UIs.
- Decoration should be decorative-only in the first slice, but the data model should allow later NetHack-like interaction with many or most decoration objects.
- Level generation should assign room functions that drive decoration and gameplay expectations, for example armory, dining hall, pantry, serving room, kitchen, reception, battlement/wall-walk, shops, and blacksmiths.
- Initial room-function selection can be creative and code-first; exact balancing can be refined after the first playable room-function slice.

## Epic: Complete SharpHack as a Playable Roguelike
Deliver a complete end-to-end game experience with a clear objective, reliable progression, meaningful choices, persistence, and polished play surfaces.

### Feature 1: Core Game Loop Completion
**Value**
Players can start a run, progress through dungeon depth, die, win, and restart without undefined behavior.

**Backlog Items**

#### Backlog Item 1.1: Define run lifecycle states
Scope:
- Introduce explicit run states such as `NotStarted`, `Running`, `PlayerDead`, `Victory`, and `Abandoned`.
- Prevent actions after terminal states.
- Surface lifecycle state through ViewModels.

Acceptance Criteria:
- Player death changes the session to a terminal state.
- Victory changes the session to a terminal state.
- Movement, wait, pickup, and door actions fail safely after terminal states.
- Tests cover state transitions.

#### Backlog Item 1.2: Define victory and dungeon-depth rules
Scope:
- Decide target depth and finale rules.
- Add a final-level objective such as a boss, artifact, or exit.
- Add scoring or completion summary data.

Product Decision:
- The final victory condition is obtaining the Amulet of JoCarneer.
- Dungeon depth progression and finale structure should support the Amulet objective instead of replacing it.

Acceptance Criteria:
- A run has a documented completion objective.
- Reaching the objective triggers victory.
- The UI can show a victory summary.

#### Backlog Item 1.3: Add restart and new-run flow
Scope:
- Add a clean session factory abstraction.
- Support creating a new run without application restart.
- Keep run setup reusable across console, WPF, terminal, and server.

Acceptance Criteria:
- Console and WPF2D can start a new run after death or victory.
- New runs do not leak state from previous runs.
- Tests cover factory-created run defaults.

### Feature 2: Durable Persistence
**Value**
Players can save and resume a run reliably.

**Backlog Items**

#### Backlog Item 2.1: Implement save-game model
Scope:
- Define serializable DTOs for run state, player, map, tiles, items, enemies, equipment, explored tiles, and level metadata.
- Avoid serializing UI-specific types.
- Include save-version metadata.

Acceptance Criteria:
- Save DTOs capture all gameplay state needed to resume a run.
- DTOs are independent from WPF and console concerns.
- Tests verify round-trip serialization of a representative session.

#### Backlog Item 2.2: Implement file-backed `GamePersist`
Scope:
- Replace `NotImplementedException` in `SharpHack.Persist.GamePersist` with durable storage.
- Keep `InMemoryGamePersist` for tests and transient play.
- Handle missing, corrupt, and unsupported save files explicitly.

Acceptance Criteria:
- A run can be saved to disk and loaded again.
- Corrupt save files return a clear failure result or exception type.
- Save tests do not rely on user-specific paths.

#### Backlog Item 2.3: Add save/load UI flows
Scope:
- Add console commands for save, load, and quit.
- Add WPF commands/dialog integration for save/load.
- Keep path selection and UI strings in UI-facing layers.

Acceptance Criteria:
- Console players can save and resume a run.
- WPF2D players can save and resume a run.
- ViewModels expose state needed by UI without direct file-dialog dependencies.

### Feature 3: Content and Progression
**Value**
The game becomes replayable through varied enemies, items, level depth, and rewards.

**Backlog Items**

#### Backlog Item 3.1: Introduce data-driven content definitions
Scope:
- Define content records for creature archetypes, item archetypes, spawn tables, and depth scaling.
- Move hard-coded goblin/sword/armor creation out of `GameSession`.
- Keep initial definitions in code or embedded data until external content format is decided.

Acceptance Criteria:
- Enemy and item spawning uses content definitions.
- Tests can inject deterministic content tables.
- Adding a new enemy does not require editing core session turn logic.

#### Backlog Item 3.2: Expand item systems
Scope:
- Add explicit equip, unequip, drop, inspect, and pickup actions.
- Add consumables such as healing potions.
- Define stack and container behavior.

Acceptance Criteria:
- Inventory operations update map and creature state consistently.
- Equipment changes affect combat stats.
- Consumables apply effects and are removed or decremented.
- Tests cover inventory edge cases.

#### Backlog Item 3.3: Add level scaling and rewards
Scope:
- Scale enemy count, enemy strength, item quality, and map features by depth.
- Define rewards for exploration, combat, and level completion.
- Decide whether the game uses experience levels, equipment-only growth, or both.

Acceptance Criteria:
- Deeper levels produce measurably different encounters.
- Progression is deterministic under controlled random seeds.
- Tests verify spawn-table selection by depth.

### Feature 4: Combat, AI, and Magic Depth
**Value**
Encounters become tactically interesting rather than simple bump combat.

**Backlog Items**

#### Backlog Item 4.1: Replace simple combat with structured combat results
Scope:
- Add a combat result model with hit, miss, damage, death, and optional effect messages.
- Support random damage ranges through injected randomness.
- Keep UI message formatting outside low-level combat where practical.

Acceptance Criteria:
- Combat results are testable without parsing strings.
- Death handling remains consistent for player and enemies.
- Existing combat tests are updated to structured assertions.

#### Backlog Item 4.2: Upgrade enemy AI
Scope:
- Use pathfinding for pursuit around walls and doors.
- Add perception rules based on field of view, distance, or sound.
- Add basic states such as idle, alerted, chasing, and fleeing.

Acceptance Criteria:
- Enemies can navigate around simple obstacles.
- Enemies do not always know the player position unless rules allow it.
- Tests cover blocked paths, visible player, unseen player, and adjacent attack behavior.

#### Backlog Item 4.3: Implement first magic slice
Scope:
- Define spell contracts in `SharpHack.Magic`.
- Add at least one player spell and one enemy or environmental spell interaction.
- Surface spell selection through ViewModels.

Acceptance Criteria:
- A spell can target a valid tile or creature.
- Invalid targets fail safely.
- Spell effects are covered by tests.

### Feature 5: Map Generation and World Structure
**Value**
Generated dungeons become reliable, connected, and interesting across many seeds.

**Backlog Items**

#### Backlog Item 5.1: Standardize active map generator
Scope:
- Decide whether `BSPMapGenerator` or `BSPRoomMazeMapGenerator` is the primary generator.
- Move generator selection into a shared setup service.
- Document generator rules and constraints.

Acceptance Criteria:
- Console and WPF2D use the same default generator.
- Generator choice is configurable for tests or debug builds.
- Documentation identifies the default generator.

#### Backlog Item 5.2: Add generated-map invariants
Scope:
- Add tests for connectivity, reachable stairs, reachable player spawn, door validity, and no impossible level transitions.
- Run tests against multiple deterministic seeds.

Acceptance Criteria:
- Generated levels always include a valid player spawn.
- Stairs are reachable from the spawn.
- Doors connect meaningful regions.
- Test failures report the seed or random sequence.

#### Backlog Item 5.3: Add themed level features
Scope:
- Add room types, traps, secret doors, shops, treasure rooms, or special encounters.
- Represent feature metadata separately from raw `TileType` when needed.

Acceptance Criteria:
- At least two distinct level feature types can spawn.
- Feature placement does not break map connectivity.
- UI can render or describe new features.

### Feature 6: UI and Player Experience
**Value**
Players understand controls, status, choices, and outcomes.

**Backlog Items**

#### Backlog Item 6.1: Add shared application orchestration
Scope:
- Extract duplicated session setup from console and WPF2D into a reusable application service or factory.
- Keep UI-specific rendering in each UI project.
- Use dependency injection where practical.

Acceptance Criteria:
- Console and WPF2D create equivalent sessions through shared setup.
- Tests can instantiate setup with substitutes.
- No UI project owns core gameplay defaults exclusively.

#### Backlog Item 6.2: Improve HUD and message surfaces
Scope:
- Show HP, attack, defense, equipped weapon, armor, depth, state, and objective.
- Improve message log capacity and categorization.
- Add help text for controls.

Acceptance Criteria:
- Players can discover all major controls in-game.
- Death and victory messages are clear.
- UI text remains in UI/ViewModel layers.

#### Backlog Item 6.3: Complete inventory UI
Scope:
- Add inventory selection in console and WPF2D.
- Support equip, drop, inspect, and use commands.
- Show item effects and quantities.

Acceptance Criteria:
- Players can manage inventory without auto-equip.
- Inventory actions are available through keyboard and WPF commands.
- Tests cover ViewModel command behavior.

### Feature 7: Test, Quality, and Release Readiness
**Value**
The project can ship and evolve safely.

**Backlog Items**

#### Backlog Item 7.1: Build smoke-test suite
Scope:
- Identify the primary solution or project set for SharpHack validation.
- Add smoke tests for run creation, movement, combat, level transition, save/load, and UI ViewModel commands.
- Keep unsupported older framework warnings ignored when newer supported targets pass.

Acceptance Criteria:
- Relevant SharpHack tests pass in one documented validation path.
- Smoke tests are deterministic.
- Validation status is recorded in release notes.

#### Backlog Item 7.2: Add diagnostics and crash handling
Scope:
- Add structured error handling around UI entry points and server sessions.
- Add optional log sinks for session events and exceptions.
- Avoid leaking sensitive local paths in user-facing messages.

Acceptance Criteria:
- Unexpected exceptions are logged.
- UI surfaces show recoverable failure messages.
- Server client failures do not crash the listener.

#### Backlog Item 7.3: Prepare packaging and user documentation
Scope:
- Document controls, gameplay objective, save/load, and known limitations.
- Define packaging targets for console, WPF2D, and optional server.
- Add version and release notes conventions.

Acceptance Criteria:
- A new user can install/run the selected distribution.
- Documentation explains how to start, play, save, win, and quit.
- Release artifacts are versioned.

## Definition of Done for Complete Game

- A player can start a new run, understand controls, explore multiple levels, fight enemies, collect and manage items, save/load progress, die, win, and restart.
- Core gameplay rules are covered by deterministic MSTest tests.
- Generated maps satisfy documented invariants across representative seeds.
- Console and the selected WPF UI share core setup and gameplay services.
- Persistence is durable and versioned.
- Release documentation explains installation, controls, objective, save/load, and troubleshooting.
- Relevant builds and tests pass, ignoring only repository-approved unsupported older-framework warnings.
