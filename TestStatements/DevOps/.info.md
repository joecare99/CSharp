# DevOps Planning Notes

This repository uses the `DevOps` directory as the local planning area for Azure-DevOps-style work preparation.

## Working Model

- Hierarchy: `Epic -> Feature -> Backlog Item -> Task`
- Team terminology prefers `Backlog Item` over `Story`
- Planning files are drafts until they are copied into Azure DevOps
- Each planning file should include scope, assumptions, open questions, and next refinement steps

## Current Repository Baseline

- Active planning starts with `E0100 - Evolve TestStatements as a Discoverable Learning Platform`
- Active features currently use the `F0100` series
- Active backlog items currently use the `B0100` series
- Active tasks currently use the `T0100` series
- Planning is based on the existing `TestStatements` code base, its related projects, and meaningful framework-level extension options
- Temporary foreign artifacts may still exist in the folder tree until manually removed and should not be treated as the active backlog baseline

## File and Folder Conventions

- Root planning directory: `DevOps`
- Main folders:
  - `DevOps/Epics`
  - `DevOps/Features`
  - `DevOps/BacklogItems`
  - `DevOps/Tasks`
- File naming pattern: `<Id>-<Short-Title>.md`
- Use stable IDs where available, for example Azure DevOps work item numbers

## Planning Rules

- Keep one primary planning item per file
- Record assumptions explicitly instead of hiding uncertainty
- Prefer thin backlog items that can later be refined into implementation tasks
- Link child items to their parent item by ID and title
- Mark unknowns as open questions instead of inventing detail
- For each major coding task, create a dedicated test-task item
- Derive new planning items from the actual repository content before inventing future work
- When planning framework evolution, prefer extensions that improve discoverability, validation, documentation, and reuse of existing examples

## Refinement Flow

1. Define the epic goal and boundaries
2. Split the epic into feature candidates
3. Derive backlog items with visible user or process value
4. Add tasks only when implementation planning starts
5. Revisit assumptions and open questions after each refinement pass
